# My TIL 시간복잡도 기준 분류

데이터 규모($N$)에 따라 어떤 알고리즘을 꺼내 써야 할지 결정하는 기준입니다.

| 시간복잡도 | 분류 | 해당 알고리즘 및 자료구조 |
| --- | --- | --- |
| **$O(1)$ ~ $O(\log N)$** | **매우 빠름** | 이분 탐색, 매개 변수 탐색, 유클리드 호제법, 세그먼트 트리, 펜윅 트리, 힙(heap), 해시, 덱/스택/큐 |
| **$O(N)$ ~ $O(N \log N)$** | **효율적** | 그리디, 슬라이딩 윈도우, 투 포인터, 두 정점 사이의 최단 경로(다익스트라), KMP, 위상 정렬, 에라토스테네스의 체 |
| **$O(N^2)$** | **보통/주의** | 구간 합(2차원), 인접 행렬 기반 그래프 탐색, 버블/삽입 정렬 |
| **$O(N^3)$ 이상** | **느림 (대용량 불가)** | 플로이드-워셜, 동적계획법(DP - 상태에 따라 다름), 배낭 문제(Knapsack) |
| **$O(2^N)$ ~ $O(N!)$** | **완전 탐색** | 백트래킹, 순열, 조합, 완전 탐색(Exhaustive Search) |

제공해주신 파이썬 고성능 컴퓨팅 및 알고리즘 최적화에 관한 상세한 내용을 바탕으로, **GitHub 위키나 기술 블로그, 사내 기술 보고서(Markdown)**에 바로 활용하실 수 있도록 가독성을 극대화하여 재구성해 드립니다.

# 시간복잡도에 관한 고찰

## 1. 개요

현대 소프트웨어 개발 환경에서 파이썬은 높은 생산성을 제공하나, **인터프리터 언어**라는 구조적 특성상 실행 속도와 메모리 효율성 측면에서 면밀한 설계가 요구된다. 본 보고서는 CPython 내부 메커니즘과 데이터 구조의 저수준 구현을 바탕으로 한 성능 최적화 전략을 분석한다.

---

## 2. 파이썬 내장 데이터 구조 분석

### 2.1 리스트(List): 동적 배열 구조

파이썬 리스트는 내부적으로 **동적 배열(Dynamic Array)**로 구현되어 객체 포인터를 연속된 메모리 공간에 저장한다.

| 연산 | 평균 복잡도 | 최악 복잡도 | 성능적 시사점 |
| --- | --- | --- | --- |
| **인덱스 접근 (`l[i]`)** | $O(1)$ | $O(1)$ | 임의 접근 시 최상의 효율 제공 |
| **요소 추가 (`append`)** | $O(1)$ | $O(n)$ | 분할 상환(Amortized) 최적화 적용 |
| **요소 삽입 (`insert(i, x)`)** | $O(n)$ | $O(n)$ | 삽입 지점 이후의 모든 요소 이동 발생 |
| **포함 확인 (`x in l`)** | $O(n)$ | $O(n)$ | 선형 탐색이 필요하며 데이터 증대 시 Set 권장 |

> **💡 핵심 팁:** 리스트의 첫 번째 요소를 삭제하거나 삽입하는 연산(`pop(0)`, `insert(0, x)`)이 빈번한 경우, 양방향 큐인 `collections.deque`를 사용하여 $O(1)$의 성능을 확보한다.

### 2.2 딕셔너리(Dict) 및 집합(Set): 해시 테이블

두 자료구조는 해시 테이블을 기반으로 하며, 평균 **$O(1)$**의 탐색 성능을 보장한다. CPython은 **개방 주소법(Open Addressing)**을 채택하여 해시 충돌을 처리한다.

* **성능 저하:** 해시 충돌이 빈번하거나 로드 팩터가 임계치를 초과할 경우 연산 복잡도가 $O(n)$까지 악화될 수 있다.
* **메서드 선택:** `s.difference(t)`는 $O(\text{len}(s))$이나, `s.difference_update(t)`는 $O(\text{len}(t))$의 복잡도를 가지므로 연산 대상의 크기에 따라 전략적으로 선택한다.

### 2.3 문자열(String) 처리의 효율성

파이썬 문자열은 **불변(Immutable)** 객체이다.

* **비효율:** `s += "a"`와 같은 반복 연산은 매번 새로운 객체를 생성하므로 $O(n^2)$의 비용을 초래한다.
* **최적화:** `"".join(list)` 방식을 사용하면 결합될 전체 길이를 사전에 계산하여 단 한 번의 메모리 할당으로 $O(n)$에 처리를 완료한다.

## 3. 인터프리터 레벨 코드 최적화

### 3.1 리스트 컴프리헨션(List Comprehension)

리스트 컴프리헨션은 단순한 가독성 확보를 넘어 바이트코드 레벨에서 고도로 최적화된 성능을 발휘한다.

| 항목 | 일반 for 루프 + append | 리스트 컴프리헨션 | 성능 우위 요인 |
| --- | --- | --- | --- |
| **변수 로드** | `LOAD_NAME` (전역/지역 검색) | `LOAD_FAST` (인덱스 접근) | 지역 변수 접근 속도 최적화 |
| **메서드 조회** | `LOAD_METHOD` (반복 호출) | `LIST_APPEND` (전용 명령) | 속성 조회 오버헤드 원천 제거 |
| **함수 호출** | `CALL_METHOD` (스택 오버헤드) | 인라인 최적화 명령 실행 | 호출 스택 관리 비용 감소 |

### 3.2 변수 접근 범위와 캐싱 전략

* **지역 변수:** 고정 크기 배열로 관리되며 `LOAD_FAST` 명령을 통해 매우 빠르게 처리된다.
* **전역 변수:** 딕셔너리 기반의 해시 조회가 수반되어 상대적으로 느리다.
* **전략:** 루프 내부에서 반복적으로 참조되는 전역 변수나 내장 함수(예: `math.sqrt`)는 루프 진입 전 **지역 변수로 할당(Caching)**하여 사용한다.

## 4. 핵심 알고리즘 및 복잡도 상세 분석

### 4.1 정렬 알고리즘: Timsort

파이썬의 표준 정렬인 `sort()`와 `sorted()`는 **Timsort**를 사용한다. 이는 병합 정렬의 안정성과 삽입 정렬의 효율성을 결합한 하이브리드 방식이다.

| 알고리즘 | 최선 | 평균 | 최악 | 특징 |
| --- | --- | --- | --- | --- |
| **Timsort** | $O(n)$ | $O(n \log n)$ | $O(n \log n)$ | 실제 데이터셋에서 최상의 성능 발휘 |
| **Quick Sort** | $O(n \log n)$ | $O(n \log n)$ | $O(n^2)$ | 피벗 선택에 따른 성능 가변성 존재 |
| **Counting Sort** | $O(n+k)$ | $O(n+k)$ | $O(n+k)$ | 데이터 범위가 제한적일 때 유효 |

### 4.2 그래프 알고리즘과 표현 방식

* **그래프 표현:**
* **인접 행렬:** 간선 확인이 $O(1)$로 빠르나, 공간 복잡도가 $O(V^2)$으로 밀집 그래프에 적합하다.
* **인접 리스트:** 공간 복잡도 $O(V+E)$를 가지며, 희소 그래프에서 순회 효율이 압도적이다.


* **최단 경로 탐색:**
* 가중치 없는 그래프: **BFS** ($O(V+E)$) 사용.
* 양수 가중치 그래프: **Dijkstra** ($O((V+E) \log V)$) 사용.
* 음수 가중치 포함: **Bellman-Ford** ($O(VE)$) 사용.

## 5. 고급 자료구조 및 DP 최적화

### 5.1 특수 목적 자료구조의 활용

* **세그먼트 트리:** 구간 쿼리와 데이터 업데이트를 모두 $O(\log n)$에 수행한다.
* **펜윅 트리(BIT):** 구현이 간결하며 메모리 효율적인 구간 합 계산을 지원한다.
* **트라이(Trie):** 문자열 접두사 탐색 시 문자열 길이 $m$에 비례하는 $O(m)$ 성능을 보장한다.

### 5.2 동적 계획법(DP) 고도화

1. **메모이제이션(Memoization):** 하향식 접근으로 필요한 상태만 계산하나, 재귀 깊이 제한을 고려해야 한다.
2. **타뷸레이션(Tabulation):** 상향식 접근으로 루프를 사용하며, 재귀 오버헤드가 없고 메모리 관리가 정밀하다.
3. **비트마스킹(Bitmasking):** 집합의 상태를 정수 비트로 압축하여 연산 속도와 메모리 효율을 극대화한다.

## 6. 실행 환경 최적화 설정

* **I/O 병목:** 대량 데이터 처리 시 `input()` 대신 `sys.stdin.readline`을 필히 사용한다.
* **재귀 제한:** 깊은 탐색이 필요한 경우 `sys.setrecursionlimit()`을 통해 스택 크기를 조절한다.
* **대안 런타임:** 연산 집약적인 작업의 경우 JIT 컴파일러를 지원하는 **PyPy** 도입을 검토한다.

## 7. 데이터 규모별 알고리즘 설계 가이드라인

파이썬 인터프리터의 성능을 고려할 때, 1초 내 처리가 가능한 연산 횟수는 약 $10^7 \sim 10^8$회로 산정한다.

| 데이터 규모 ($n$) | 허용 시간 복잡도 | 해당 알고리즘 예시 |
| --- | --- | --- |
| $n \le 20$ | $O(2^n)$ | 비트마스킹 DP, 브루트 포스 |
| $n \le 500$ | $O(n^3)$ | 플로이드-워셜 알고리즘 |
| $n \le 5,000$ | $O(n^2)$ | 단순 2중 루프, 기본 DP |
| $n \le 10^5$ | $O(n \log n)$ | 표준 정렬, 트리 연산, 세그먼트 트리 |
| $n \le 10^6$ | $O(n)$ | BFS/DFS, 그리디, 스택 활용 연산 |
| $n \ge 10^8$ | $O(\log n)$ | 이분 탐색, 수학적 공식 활용 |

---

**결론:** 파이썬에서의 성능 최적화는 언어적 추상화 아래 숨겨진 **내부 구현 메커니즘을 정확히 파악**하는 것에서 출발한다. 데이터 규모에 최적화된 자료구조와 알고리즘을 선택하고 인터프리터 친화적인 코드를 작성함으로써, 파이썬의 생산성과 시스템의 고성능을 동시에 확보할 수 있다.
