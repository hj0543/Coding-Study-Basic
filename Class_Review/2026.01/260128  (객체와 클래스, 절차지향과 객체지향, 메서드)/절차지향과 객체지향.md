
# 📝 절차 지향 vs 객체 지향 (Procedural vs OOP)

프로그램을 **'어떤 관점'**으로 바라보고 설계할 것인가에 대한 차이이다.

## 1. 핵심 비교

가장 큰 차이는 **데이터(변수)와 함수(기능)를 어떻게 관리하는가**에 있다.

| 구분 | 절차 지향 (Procedural) | 객체 지향 (OOP) |
| --- | --- | --- |
| **핵심 질문** | **"어떤 순서로 처리할 것인가?"** | **"누가(어떤 객체가) 처리할 것인가?"** |
| **단위** | 함수 (Function) 중심 | 객체 (Object) 중심 |
| **구조** | 데이터와 함수가 **분리됨** | 데이터와 메서드가 **하나로 묶임** |
| **흐름** | 순차적인 실행 흐름이 중요 | 객체 간의 상호작용(메시지)이 중요 |

---

## 2. 절차 지향 프로그래밍 (Procedural Programming)

물이 위에서 아래로 흐르듯, 정해진 순서대로 데이터를 처리하는 방식이다.

### 특징

* **변수와 함수의 분리**: 데이터를 저장하는 변수와, 그 데이터를 처리하는 함수가 별개로 존재한다.
* **순차적 처리**: 명령어의 실행 순서와 흐름이 프로그램의 핵심이다.

### 코드 예시 (데이터와 기능이 남남이다)

```python
# [데이터] 변수
name = 'Alice'
age = 25

# [기능] 함수
def introduce(n, a):
    print(f"저는 {n}, {a}살입니다.")

# 데이터를 함수에 직접 전달해야 함
introduce(name, age)

```

### 한계점

* **유지보수의 어려움**: 프로그램이 커지면 데이터가 어디서 수정되는지 추적하기 힘들다. (일명 스파게티 코드)
* **전역 변수 문제**: 여러 함수가 같은 데이터를 공유하다가 예기치 않은 오류가 발생할 수 있다.

---

## 3. 객체 지향 프로그래밍 (Object-Oriented Programming)

세상의 모든 사물(객체)이 각자의 역할과 책임을 가지고 상호작용하는 것처럼 프로그램을 설계하는 방식이다.

### 특징

* **데이터와 메서드의 결합**: 관련된 변수(속성)와 함수(메서드)를 **객체**라는 하나의 단위로 묶어서 관리한다.
* **설계도(Class)와 제품(Instance)**: 클래스라는 틀을 만들고, 이를 통해 인스턴스를 생성하여 사용한다.

### 코드 예시 (데이터와 기능이 한 몸이다)

```python
class Person:
    def __init__(self, name, age):
        self.name = name # [데이터]
        self.age = age

    def introduce(self): # [기능]
        # 자기 자신의 데이터를 직접 사용 (인자 전달 불필요)
        print(f"저는 {self.name}, {self.age}살입니다.")

alice = Person('Alice', 25)
alice.introduce() # 객체에게 "소개해!"라고 명령(메시지)을 보냄

```

## 4. 오해와 진실

> **"절차 지향은 나쁘고 객체 지향은 좋은 것인가?" (X)**

* **상호보완적 관계**: 객체 지향은 절차 지향을 완전히 버린 것이 아니다. 객체 내부의 메서드 로직은 여전히 절차 지향적으로(순서대로) 돌아간다.
* **발전 과정**: 프로그램의 덩치가 커지고 복잡해지면서, 기존 절차 지향의 한계(유지보수 어려움)를 해결하기 위해 **데이터 관리 방식을 효율화(객체화)**한 진화된 형태이다.
